# Preset Re-Design Proposal — Time-Budget Pool Selection

Status: DRAFT — pending Codex review.

## Problem statement

The current 4 presets in `bench/scenarios/{fast_reliable,strong_practical,best_current,exhaustive_research}.json` were designed as a **publication-rigor progression** (gated by `protocol_maturity`, `inner_cv_nested`, `supports_predefined_test_split`, Codex review verdicts). They are essentially nested AOM-Ridge stacks.

The user's stated requirement is different: provide a **time-budget pool** so an end user can pick a preset based on their compute budget and get the best score from an unknown dataset.

> "Le user a 20 sec, il prend le fast_reliable, il a plus il prend le strong_practical, etc. … TabPFN HPO qui prend beaucoup de temps mais qui a le plus de chance de fournir le bon score est dans le preset le plus cher et peut-être aussi l'avant-dernier mais pas dans le preset rapide."

The current presets explicitly **exclude TabPFN from every tier** (per the registry comments: "TabPFN family deferred from Phase 1 production runs"). They are not time-budget pools.

## Proposed methodology

1. Aggregate `bench/benchmark_master_results.csv` to obtain, per `(canonical_name, dataset)`, the best `rmsep` observed. Maturity = locked or exploratory both allowed (we want diversity).
2. For every dataset, take `PLS-tuned-cv5` rmsep as baseline.
3. Compute `ratio_m,d = best_rmsep_m,d / pls_d`, clip at 5×.
4. Per canonical name: `median_ratio`, `q75_ratio`, `q90_ratio`, `wins_vs_pls`, `n_datasets` of evidence, median + q90 fit_time.
5. For each preset, define a per-dataset *sequential* wall-clock budget `B`:
   - fast_reliable: 30 s (user said ~20 s)
   - strong_practical: 600 s (10 min)
   - best_current: 7200 s (2 h, per user 2026-05-12)
   - exhaustive_research: 43 200 s (12 h, per user 2026-05-12)
6. **Quality gates per preset** (hard filters on candidate set):

| preset | max q90_ratio | min wins/n_ds | min latest_run_success_rate | max pool size |
|---|---:|---:|---:|---:|
| fast_reliable | 1.50 | 0.30 | 0.90 | 5 |
| strong_practical | 1.80 | 0.20 | 0.50 | 8 |
| best_current | 2.50 | 0.15 | 0.30 | 12 |
| exhaustive_research | 5.00 | 0.05 | — | 20 |

7. **Greedy pool selection**: starting from `[PLS-tuned-cv5]`, repeatedly add the model that minimises `mean over datasets of min_{m∈pool}(ratio_m,d)` subject to `Σ_m q90_fit_time_m ≤ B`, stopping when no improvement or the pool size cap is reached.
8. **TabPFN fit-times** come from the v2.5 paper because master CSV TabPFN rows are predict-only:
   - TabPFN-Raw: median 3s / q90 10s
   - TabPFN-opt: median 60s / q90 180s
   - TabPFN-HPO-preprocessing: median 900s / q90 1800s

## Areas Codex should scrutinise

1. **Objective function**: is `mean over datasets of min-of-pool ratio` the right surrogate for the user's stated criterion? Alternatives: `median-of-min`, `q90-of-min`, or `EXPECTED_BEST = E_d[min_m ratio]` weighted by dataset importance.
2. **Time budget choice**: 30 / 600 / 7200 / 43 200 s — defensible? The user said "20 s" for the cheapest; 30s gives one safety margin step. Budgets clarified by user 2026-05-12: best=2h, exhaustive=12h.
3. **TabPFN external times**: numbers are from the v2.5 paper. Are they realistic on the user's RTX 4090?
4. **POP-PLS / V2L-Boost inclusion**: these have catastrophic q90 (4.97, 2.88) but win on specific datasets. The gates exclude them from non-research presets but they appear in exhaustive_research. Is that the right trade-off?
5. **0% runnability models** (MKM-reml-default, mkR-softmax-cv-default, moe-preproc-soft-pls-compact, Stack-Ridge-PLS-V1c) failed all 57 latest production attempts but have master CSV evidence. Should they appear in proposed exhaustive_research? Current code keeps them because evidence is older but valid.
6. **TabPFN gating**: registry says `max_n=5000, max_features=1000`. The audit doesn't enforce this at preset definition; it's assumed the harness checks at dispatch time. Is that OK?
7. **Pool size caps**: 5/8/12/20. Are these reasonable for the harness scheduler?
